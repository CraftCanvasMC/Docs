:::note

This guide assumes you are able to use a Gradle wrapper of some capacity, either Gradle Kotlin DSL
or Gradle Groovy DSL. We do not include guides for accesing internals on Maven.
However, should you wish to access Canvas' internals with Maven, there are community made plugins [available](https://github.com/Alvinn8/paper-nms-maven-plugin).

This page focuses on teaching you how to access internals through the `weaver-userdev` Gradle plugin.
But given that `paperweight-userdev` also allows you to access internals, we provide guides for using it too.

You might also find some additional resources, going more in-depth on the [PaperMC Userdev Docs](https://docs.papermc.io/paper/dev/userdev/).
:::

## What is weaver, and how does it differ from paperweight?

[**weaver**](https://github.com/CraftCanvasMC/weaver) is a custom fork of [**paperweight**](https://github.com/PaperMC/paperweight) used for Canvas build tooling.
It includes 3 modules, but only one of them matters here, namely **weaver-userdev**.
The main difference between the `paperweight` userdev plugin and ours is that **weaver** has some DSL extensions that make using Canvas dev bundles more easy.

## What are `dev bundle`'s?

Dev bundles are maven artifacts, containing patches to the Minecraft source, which are applied during the build process, and API.
This means that they are a fully functional way to access Minecraft internals, without having to worry about EULA or license restrictions.

## Is This Important to You?

Accessing server internals is not necessary for majority of Canvas plugins.
Using the Canvas API or the one of the APIs of our upstream projects (i.e. Paper or Folia) is always preferrable when it can be used.

In some situations, the Canvas API may not be enough and you may need to access server internals directly.

## Getting Started

:::danger

The Canvas API should **always** be preferred over internals.
Minecraft and Canvas internals change *very* frequently and your plugin **is likely to break** upon every new Minecraft release.
If your code can be written using Canvas' API, [use the API](/canvas/developers/canvas-api/)!

If you're not sure whether your plugin requires access to Minecraft internals, join our [Discord](https://canvasmc.io/discord)!
We're happy to give feedback and help you where we can.

:::

If you've concluded that your project has to rely on internals, this guide is here to assist you in configuring your Gradle build script, so that you can build against them.

### Setup using `weaver-userdev`

This portion will show you how to add **weaver-userdev** and the Canvas dev bundle to your Gradle project.

In your `settings.gradle.kts` or `settings.gradle` file...

```kotlin title="settings.gradle.kts"
pluginManagement {
    repositories {
        gradlePluginPortal()
        maven("https://maven.canvasmc.io/public")
    }
}
```

In your `build.gradle.kts` or `build.gradle` file...

```kotlin title="build.gradle.kts"
plugins {
    id("io.canvasmc.weaver.userdev") version "2.4.3"
}

dependencies {
    // Other dependencies...
    paperweight.canvasDevBundle("26.1.2.build.+")
}

java {
    toolchain.languageVersion.set(JavaLanguageVersion.of(25))
}
```


### Setup using `paperweight-userdev`

:::note

This is an alternative method for adding Canvas dev bundles to your Gradle project.
`paperweight-userdev` is not developed, nor maintained by CanvasMC. [**weaver-userdev**](/canvas/developers/project/userdev/#weaver-userdev-setup) should always be preferred!

:::


```kotlin title="build.gradle.kts"
plugins {
    id("io.papermc.paperweight.userdev") version "2.0.0-beta.21"
}

repositories {
    // Other respositories...
    maven("https://maven.canvasmc.io/public")
}

dependencies {
    // Other dependencies...
    paperweight.devBundle("io.canvasmc.canvas", "26.1.2.build.+")
}

java {
    toolchain.languageVersion.set(JavaLanguageVersion.of(25))
}
```

After completing all of the above steps, you will have successfully attached Minecraft internals to your project!
